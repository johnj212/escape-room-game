# R3F + WebGPU + TSL — verified stack notes

Append-only log of **verified** facts about the render/stack surface this build depends on.

**Rule:** before relying on an unfamiliar API (three.js `WebGPURenderer`, TSL node, R3F v9 behavior, Rapier, WGSL compute), verify it against the **actually installed source** in `node_modules` — not training-data memory — and record what you found here with the version you checked. Memory of these APIs is stale and they change between minor versions.

---

## Seed facts (confirm against installed versions before relying on them)

- **WebGPU needs React Three Fiber v9.** The current build is pinned to Fiber v8 + `@react-three/postprocessing` v2 (see `handoff.md` §3D) — that pin is retired in Phase 1. Verify the v9 `<Canvas>` → `WebGPURenderer` wiring (frameloop, `gl` factory / async init) against the installed `@react-three/fiber` source before building on it.
- **Rapier is renderer-agnostic.** Physics runs on WASM/CPU independent of WebGPU vs WebGL — the D-1 physics choice is unaffected by the renderer pivot.
- **`WebGPURenderer` init is async** (device/adapter request). The capability gate must `await` adapter acquisition and route to the unsupported screen on failure — never assume a synchronous context like WebGL.
- **TSL is the primary shading path; raw WGSL compute is the escape hatch.** Expect to drop to WGSL for the irradiance-probe bake and any pass TSL can't express. Record which passes ended up in raw WGSL and why.

## WebGPU availability (as of mid-2026 — re-verify targets)

- Desktop: Chrome/Edge 113+, Safari 18+, Firefox 141+ (Windows). Broadly safe.
- Mobile: iOS 18+ Safari; Android Chrome 121+ with a Vulkan-capable GPU. **iOS ≤ 17 and budget/older Android have no WebGPU** → these hit the designed unsupported screen (no fallback renderer).

## Verified findings

_(Add entries as APIs are checked against installed source, newest first. Format: `- <date> [pkg@version] <what was verified / gotcha>`.)_

- 2026-07-07 [three@0.185.1 TSL display nodes — post-stack build] **Three gotchas from wiring `PostProcessing` + TSL display nodes** (each produced a silently-wrong frame, zero console errors):
  - **`chromaticAberration()` only works on a real texture node** (it re-samples per channel via `textureNode.sample(uv)`). Feeding it a computed chain (`beauty.add(bloomPass)`) breaks the G/B taps. Apply CA directly to `scenePass.getTextureNode('output')`; `bloom()` by contrast renders its input into internal targets, so computed chains are fine there. Also `chromaticAberration`'s default `center = null` is consumed unconditionally → NodeBuilder crash ("Cannot read properties of null (reading 'build')"); always pass `vec2(...)`.
  - **GTAO's output target is single-channel** — `aoPass.getTextureNode()` samples as `(ao, 0, 0, 1)`. Component-wise `.mul(color)` zeroes G/B and the whole frame collapses to red. Multiply by the scalar: `color.mul(aoPass.getTextureNode().r)`.
  - **Full-res GTAO at dpr 2 ran ~1 fps** on Apple M3 headless (2880×1620, 16 samples). `aoPass.resolutionScale = 0.5` + `aoPass.samples.value = 8` restored 60 fps with near-identical read. R3F integration: a priority-1 `useFrame(() => post.render(), 1)` suppresses fiber's own render (`state.internal.priority`), and `renderer.info` keeps feeding PerfProbe.

- 2026-07-07 [react@19.2.7, @react-three/fiber@9.6.1, three@0.185.1 — now INSTALLED in client/] **Three hard-won gotchas from the Phase-1 renderer swap** (each cost a debugging round; do not rediscover):
  - **React 19 `<StrictMode>` kills the fiber frame loop under an async WebGPU `gl` factory.** StrictMode's mount→unmount→remount races the awaited factory; the global loop cancels against a stale root (guard values all look correct: `frameloop:'always'`, `active:true`, `frames:1`) and the scene freezes on its first frame with **zero errors**. Empirically bisected: removing StrictMode → loop runs 60 fps. StrictMode is therefore OFF in `client/src/main.jsx` (documented there). Diagnostic that found it: spy on `requestAnimationFrame` and count `function loop(timestamp)` callbacks — 20 ticks then silence, while three's own Animation rAF kept running at 60/s.
  - **drei `<ContactShadows>` is WebGL-only in practice**: it customizes `MeshDepthMaterial` via hooks the WebGPU NodeBuilder rejects — `THREE.NodeBuilder: Material "MeshDepthMaterial" is not compatible` spams every frame. Removed from `Room.jsx`; contact grounding returns with the real §2 lighting rig (PCSS + screen-space contact shadows).
  - **Never read `renderer.info` from `useFrame` under WebGPU.** `WebGPURenderer` runs an internal Animation rAF loop that calls `info.reset()` every tick (`three/src/renderers/common/Animation.js:75`), so pre-render reads race the reset and report zeros. Read stats in fiber's `addAfterEffect` (runs synchronously after `gl.render()` in the same callback). Also: per-frame draws are `info.render.drawCalls` under WebGPU; `info.render.calls` is cumulative. **This bug also invalidated the recorded WebGL "baseline" draw/triangle numbers** (1 call / 2 tris — sampled mid-reset); the WebGL fps (27) and bundle (1083 KB gzip) numbers were real.
  - Migration facts that just worked: `lucide-react` needed a major bump for React 19 (0.378 → 1.x); rapier 2.2.0 required no code changes (`RigidBody`/`CapsuleCollider`/`linvel` API stable); drei `Html`, `useProgress`, `AdaptiveDpr/Events` fine under WebGPU; drei `Environment preset` was removed anyway (fetches an external HDR at runtime — Pillar B violation independent of the renderer).

- 2026-07-07 [@react-three/fiber@9.6.1, three@0.185.1, scratch install — NOT yet in client/] **R3F v9 + `WebGPURenderer` integration path confirmed against installed source** (Phase-0 next-action #4):
  - **The `gl` prop accepts an async factory and fiber awaits it.** Verified in fiber's dist (`events-*.esm.js`, `configure()`): `const customRenderer = typeof glConfig === 'function' ? await glConfig(defaultProps) : glConfig`, accepted when `isRenderer(customRenderer)` — and `isRenderer` is just `def => !!(def != null && def.render)`, so `WebGPURenderer` qualifies. The canonical wiring is therefore: `<Canvas gl={async (props) => { const r = new WebGPURenderer({ ...props }); await r.init(); return r }}>` — the factory itself must `await r.init()` (confirmed `WebGPURenderer.prototype.init` exists; fiber does NOT call init for you, it only awaits the factory).
  - **Fiber 9.6.1 requires React 19** (peer: `react >=19 <19.3`). Phase 1's dep bump is therefore also a React 18.3 → 19 migration.
  - **Phase-1 dependency matrix** (all latest, all peer-locked to fiber ^9 + react ^19): `@react-three/drei@10.7.7`, `@react-three/rapier@2.2.0` (fiber ^9.0.4), `@react-three/postprocessing@3.0.4`. **But note:** `@react-three/postprocessing` wraps the WebGL-only `postprocessing` library — under `WebGPURenderer` the post stack should instead use three's own `PostProcessing` class (exported from `three/webgpu`, verified) with TSL display nodes. Plan to drop `@react-three/postprocessing` entirely in Phase 1.
  - **`three@0.185.1` export map**: `three/webgpu` (verified `WebGPURenderer`, `PostProcessing`) and `three/tsl` (638 exports, incl. `pass`). First-party TSL display nodes for essentially the entire §2 post/GI stack exist under `three/examples/jsm/tsl/display/`: `BloomNode` (verified `bloom` import), `GTAONode`, `GodraysNode` (volumetric shafts), `ChromaticAberrationNode`, `DepthOfFieldNode`, `FXAANode`, `Lut3DNode`, `FilmNode` — Phase 1/5 should build on these rather than hand-rolling.
  - Scratch install lives outside the repo (session scratchpad `r3f9-probe/`); `client/` remains on the v8 stack until the Phase 1 swap.

- 2026-07-04 [@playwright/test@1.61.1, playwright chromium v1228 = Chrome-for-Testing 149.0.7827.55, macOS 26.5.2 / Apple M3] **Empirically verified Playwright WebGPU launch flags** (Phase 0 harness work, done ahead of the Phase 1 renderer rewrite so Phase 1 doesn't have to re-derive this):
  - `navigator.gpu` does **not exist at all** on an opaque-origin page (`about:blank`) or any page where `window.isSecureContext === false`. Always `page.goto()` a real `http://localhost:*` or `https://` origin before probing/using WebGPU — this is a browser security requirement, not a Playwright limitation.
  - On a real origin, headless Chromium (`headless: true`, the Playwright default "new" headless mode) exposes `navigator.gpu` but `requestAdapter()` resolves to `null` with **zero launch args** — no adapter.
  - Passing `--enable-unsafe-webgpu --use-angle=metal` as Chromium launch args makes headless Chromium return a **real, non-null, hardware-backed adapter** using the actual Metal backend on Apple Silicon — verified by comparing `adapter.features`/`adapter.limits` against a headed (non-headless) run with no flags: both report identical feature sets (`shader-f16`, `texture-formats-tier1`/`tier2`, `primitive-index`, `maxTextureDimension2D: 16384`), confirming it is not a SwiftShader/software fallback.
  - `--use-angle=metal` alone (without `--enable-unsafe-webgpu`) also produced a working hardware adapter on this Chrome version; `--enable-unsafe-webgpu` is kept anyway as the documented/stable cross-version way to request WebGPU in Chromium.
  - Flags that made **no difference** either way (adapter was null without the two flags above regardless): `--enable-features=WebGPU`, `--ignore-gpu-blocklist`, `--disable-field-trial-config`, `--use-gl=angle` (without `--use-angle=metal`), `--headless=old` vs `--headless=new`.
  - Verification script and full flag matrix were run ad hoc during Phase 0 harness setup (not committed — the durable result is `client/playwright.config.js`'s `WEBGPU_ARGS` plus this note and `client/e2e/puzzle1.spec.js`'s "WebGPU harness capability" test, which re-verifies this on every suite run).
  - **Gotcha for Phase 1**: the current app (`client/src/components/GameCanvas.jsx`) still uses a stock R3F v8 `<Canvas>` → `WebGLRenderer`. Nothing in `client/src` requests WebGPU yet — the renderer rewrite is still Phase 1, unstarted as of this note. Don't assume the "WebGPU-only" end-state described in `Project_Requirements.md` is already true of the checked-in build.
